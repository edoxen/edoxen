# 15 — Arch: investigate extracting `schema/_shared.yaml`

## Symptom

`schema/edoxen.yaml` (69 $defs) and `schema/meeting.yaml` (59 $defs)
intentionally re-declare a shared set of $defs so each file is
self-contained. The current cross-file sync spec enforces
byte-for-byte equality across the shared set.

Today the duplication is **managed** (the spec catches drift) but the
structural cost is real:

- Adding a new shared $def requires editing both files + adding to
  `SHARED_NAMES`.
- Schema reviewers can't pattern-match on a single source.
- A maintainer who removes a $def from one file but not the other gets
  a sync-spec failure — fixable, but noisy.

## Options

**A. Extract `schema/_shared.yaml`** with the shared $defs only; both
top-level schemas `$ref` into it via `"$ref":
"_shared.yaml#/$defs/Person"`. Requires a custom `ref_resolver` in
`SchemaValidator` (json_schemer doesn't resolve cross-file refs by
default). Net: smaller files, one source per shared $def, but the
validator gets more complex.

**B. Generate one schema from the other.** Have `meeting.yaml` be the
canonical source and `edoxen.yaml` be generated by a build script. Net:
one source, but adds a build step.

**C. Status quo + better specs.** Keep the duplication, but add a
"discovery" spec that finds shared $defs automatically and asserts
they're in `SHARED_NAMES`. (TODO 07 already does this.) Net: zero
production change, drift stays caught.

## Decision

Defer (A) and (B). The structural cost of duplication is bearable, the
sync spec catches drift, and adding a `ref_resolver` to
`SchemaValidator` violates the file's "intentionally small" mandate.
If the shared set grows past ~40 $defs, revisit.

This TODO captures the analysis so future maintainers don't re-litigate.

## Acceptance

This TODO is itself the acceptance. No code change.
